Chapter 1: Welcome to Claude Code

What Is Claude Code?

Claude Code is an AI pair programmer that lives directly in your terminal (that black window where you type commands). Instead of writing code yourself, you describe what you want in plain English, and Claude writes the code, edits files, runs commands, manages your projects, and even deploys them — all while you watch.

Think of it like having a senior software developer sitting next to you 24/7, who never gets tired, never forgets context, and can read your entire codebase in seconds.

💡
Your Setup: You are running Claude Opus 4.6 with a 1 million token context window. That means Claude can read and understand roughly 750,000 words of code at once — equivalent to about 15 full novels. Your entire project history, every file, every configuration — it all fits.

  • Deploy with complex terminal commands

⚡ With Claude Code

  • Claude searches docs and the web for you

  • /deploy — one command to production
  • Everything happens in one conversation

What Makes It Different from ChatGPT or Copilot?

ChatGPT gives you code snippets you have to copy-paste into the right files yourself. It cannot see your project or run commands.

GitHub Copilot autocompletes single lines inside a code editor. It helps with typing, not thinking.

Claude Code lives inside your project. It can read every file, understand how they connect, edit multiple files at once, run your application, check if things work, search the internet, manage your GitHub repos, post to social media, and deploy your code — all from one conversation.

It does not just suggest code. It does the work.

Chapter 2: The Basics — Talking to Claude

How to Start a Conversation

When you open Claude Code, you see a text prompt. Just type what you want, like sending a message. There is no special syntax or programming language to learn. Claude understands natural language.

$ claude

Welcome to Claude Code! How can I help you today?

> I need a landing page for my removal company with a booking form

That is it. Claude will ask clarifying questions if needed, then get to work.

The Art of Good Prompts

The quality of what Claude builds depends heavily on how well you describe what you want. Think of it like giving instructions to a builder — "make it nice" is not as helpful as "use oak flooring, white walls, and brass fixtures."

Infographic: The Prompt Quality Spectrum
🔴 Vague 🟠 Decent 🟢 Precise
"Fix the bug"
Claude has to guess everything
"Fix the login bug"
Better, but still ambiguous
"The login form returns 401 with valid credentials. Check JWT middleware in auth.py"
Claude goes straight to the fix

Real Prompt Examples

Example 1: Fixing a Bug

❌ Vague Prompt
"Fix the login bug"
✅ Precise Prompt
"The login form on /auth/login returns a 401 error even with valid credentials. Check the JWT middleware in auth.py — I think the token expiry might be wrong."

Example 2: Improving Design

❌ Vague Prompt
"Make it look better"
✅ Precise Prompt
"Update the hero section to use a purple-to-cyan gradient background with glassmorphism cards, matching the Rick.AI brand. Use Bricolage Grotesque for headings."

Example 3: Adding a Feature

❌ Vague Prompt
"Add social media"
✅ Precise Prompt
"Add an Instagram feed section to the Magna Park homepage that shows the 6 most recent posts. Pull from the Instagram API using the existing social media MCP server."

When to Be Specific vs When to Let Claude Explore

Be specific when:

  • You know exactly what you want (design specs, specific bugs, file paths)
  • You have a particular style or brand to match
  • There is a technical constraint Claude needs to know about

Let Claude explore when:

  • You are starting a new project and want best practices
  • You are not sure what the best approach is
  • You want Claude to audit existing code and find issues
🚀
Pro tip: You can always start broad and refine. Say "Build a booking form for Astra Removals" and then follow up with "Make the date picker use a calendar widget instead of a text input."

When to Use Each Command

🏗️ /bootstrap — Starting a Brand New Project

Use this when you have a blank folder and want the full treatment. It creates everything:

  • CLAUDE.md with project identity and agent rules
  • tasks/ folder with todo.md and lessons.md
  • Git repository initialised locally
  • GitHub remote repository created
  • Environment configuration
  • Project scaffolding based on your tech stack

Real example: When you started the transport-manager project, /bootstrap set up the entire FastAPI project structure, database models, and deployment config in one go.

🚀 /deploy — Shipping to Production

Use this when you are ready to go live. Claude will:

  1. Run pre-deploy checks (tests, linting, build)
  2. Push your code to GitHub
  3. Monitor the build process on Vercel/Railway
  4. Run a health check on the live URL
  5. Report back with a summary

If anything fails, Claude tells you what went wrong and offers to fix it.

🔎 /audit — Project Health Check

Run this periodically to catch problems before they bite. It checks:

  • Test coverage and failing tests
  • Missing or outdated environment variables
  • Security vulnerabilities in dependencies
  • Code quality issues
  • Dependency updates available

Real example: Running /audit on primehaul-leads caught three missing env vars that would have caused crashes in production.

📡 /social-onboard — Full Social Media Setup

This is the fully autonomous version. Claude does everything itself and only interrupts you when it needs you to:

  • Log into a social media platform in the browser
  • Approve app permissions
  • Copy an access token

It batches questions together to minimise interruptions. Everything else — configuring accounts, testing connections, verifying posts — is handled automatically.

Chapter 5: Project Management with Claude

The CLAUDE.md File — Your Project's Brain

Every project you work on has a special file called CLAUDE.md at its root. This is like a briefing document that tells Claude everything it needs to know about the project: what it is, how it works, what the rules are, and how you like things done.

When Claude opens any project, the first thing it reads is CLAUDE.md. This is why Claude already knows about your projects, your preferences, and your tech stack without you having to explain every time.

Infographic: Anatomy of a Project
your-project/
├── CLAUDE.md ← Project brain — identity, rules, tech stack
├── tasks/
│   ├── todo.md ← Current tasks and progress
│   └── lessons.md ← Patterns learned from mistakes
├── src/ ← Your application code
├── .env ← Secret keys and config
├── package.json ← Dependencies
└── ...

What Goes in CLAUDE.md?

Identity: Project name, one-line description, owner (you!), tech stack.

Agent Behaviour Rules: How Claude should work — plan before building, verify before marking done, use sub-agents for research, update lessons after mistakes.

Core Principles: Simplicity first, find root causes, senior-level code standards.

Project-Specific Notes: API keys to use, deployment targets, brand guidelines, special requirements.

💡
You never need to create CLAUDE.md manually. Your global rules automatically create one when Claude enters a new project directory. If you want to add project-specific notes, just tell Claude: "Add to CLAUDE.md that this project uses Supabase for auth."

Task Management: todo.md

The tasks/todo.md file is where Claude tracks what needs to be done, what is in progress, and what is complete. This persists between conversations, so when you come back the next day, Claude picks up exactly where it left off.

# Todo

## In Progress
- [ ] Add booking calendar to Astra Removals homepage
- [ ] Fix mobile nav overlap on coach-mentor

## Done
- [x] Set up social media automation for Magna Park
- [x] Deploy transport-manager to Railway
- [x] Create PDF quote generator for primehaul-survey

The Self-Improvement Loop: lessons.md

Whenever Claude makes a mistake and you correct it, it writes down what went wrong in tasks/lessons.md. Next time, it checks this file first and avoids repeating the same error. This means Claude literally gets better at working with your projects over time.

# Lessons Learned

## 2026-03-15 — Vercel Edge Functions
- Pattern: Large dependencies break edge function size limits
- Solution: Use dynamic imports and tree-shake unused modules
- Rule: Always check bundle size before deploying to Vercel

## 2026-03-10 — Supabase RLS
- Pattern: Forgetting Row Level Security policies causes 403 errors
- Solution: Always create RLS policies when adding new tables
- Rule: Never create a Supabase table without an RLS policy

Chapter 6: Memory — How Claude Remembers You

Claude Code has a persistent memory system. This is not like ChatGPT where every conversation starts fresh. Claude remembers who you are, what projects you are working on, how you like things done, and what has happened in previous conversations.

Infographic: Memory Types
🧑
User Memories
Who you are, your name (Ricky), your email, your role as a business owner. Applies across all projects.
💬
Feedback Memories
How you like to work — "please do not ask for human permissions", your preferred code style, design preferences (dark theme, glassmorphism).
📋
Project Memories
What each project is, its current status, tech stack, deployment target, and recent changes. Stored in MEMORY.md and project CLAUDE.md files.
🔗
Reference Memories
Where to find things — shared modules location, slash command paths, MCP server configurations, deployment URLs.

How Memory Persists

Memory is stored in two places:

Global Memory (~/.claude/projects/.../MEMORY.md): Your identity, preferences, and cross-project knowledge. This follows you everywhere.

Project Memory (CLAUDE.md + tasks/): Project-specific context. This stays with the project and is read whenever Claude enters that directory.

There is also a PROGRESS.md file that Claude checks at the start of every session to pick up where you left off. This is like leaving a note for yourself about exactly what you were doing when you stopped.

Managing Your Memory

💡

To make Claude remember something:

"Remember that I always want dark theme with purple/cyan gradients for Rick.AI projects."

To check what Claude remembers:

"What do you know about my projects?"

To update memory:

"Update the Magna Park project status to deployed."

Chapter 7: Advanced Techniques

For complex tasks, Claude can launch sub-agents — think of them as assistants that each handle one piece of a bigger job. This is like having a team instead of a single person.

⚡
Trigger parallel work by saying: "Do all of these in parallel" or "Handle these simultaneously." Claude will launch multiple operations at once when they do not depend on each other.

Plan Mode

For tasks with three or more steps, Claude automatically enters plan mode. Instead of diving straight in, it outlines what it intends to do and waits for your approval. This prevents wasted work on misunderstood requirements.

> Add a complete booking system to Astra Removals with
  calendar, quote calculator, and email confirmation.

Claude responds:
  "Here's my plan for the booking system:
   1. Create BookingForm component with date/time picker
   2. Build quote calculator (distance x volume x date)
   3. Add email confirmation via SendGrid API
   4. Create booking confirmation page
   5. Add admin dashboard for managing bookings
   6. Write tests for the calculator logic

   Shall I proceed with this plan?"

You can approve the plan, modify it ("skip step 6 for now"), or ask Claude to rethink ("I want Resend instead of SendGrid").

Autonomous Mode

When you trust Claude to handle a task end-to-end without checking in, give it full permission:

> You have full permission. Build the entire landing page for
  Magna Park Self Storage. Use the Rick.AI dark theme. Deploy
  when done. Do not ask for human permissions.

Claude will work through the entire task autonomously — creating files, writing code, testing, committing to git, and deploying — only stopping if it hits an actual error it cannot resolve.

⚠️
Use autonomous mode when: You know what you want and trust the outcome. Stay hands-on when: You are experimenting, working with sensitive data, or the task is ambiguous.

Git & GitHub Integration

Claude handles all your version control. You never need to remember git commands:

"Commit this work" — Claude stages the right files, writes a clear commit message, and creates the commit.

"Create a PR for this feature" — Claude pushes to GitHub, creates a pull request with a proper title and description, and gives you the URL.

"What changed since yesterday?" — Claude shows you a summary of recent commits and changes.

"Review the code in this PR" — Claude reads the PR changes and gives you a detailed code review with suggestions.

Chapter 10: Tips & Best Practices

These ten practices will help you get the most out of Claude Code every day.

Be Specific About What You Want

The more detail you give, the closer the first attempt is to what you imagined. Include file paths, colours, feature names, and expected behaviour. "Add a purple CTA button below the hero that says 'Get a Quote'" beats "add a button" every time.

Give Context — File Paths and Error Messages

When something is broken, paste the error message directly. Mention the file or page where the issue appears. Claude can read hundreds of files instantly, but knowing where to start saves time.

Use Slash Commands for Common Tasks

Do not re-explain what "deploy" means every time. Just type /deploy. The slash commands are purpose-built automations that handle multi-step processes in one go.

Let Claude Read Code Before Editing

Claude always reads files before modifying them, ensuring it understands the existing code structure. If you notice Claude is about to make a change that conflicts with existing code, say "read the file first" to make sure it has the full picture.

Say "Full Permission" for Autonomous Work

When you want Claude to handle everything end-to-end without asking for approval at each step, say "you have full permission" or "do not ask for human permissions." This unlocks autonomous mode for maximum speed.

Use "In Parallel" for Independent Tasks

If you need multiple things done that do not depend on each other, say "do these in parallel." Claude will launch multiple operations simultaneously, cutting total time significantly.

Review Plans for Complex Changes

Claude enters plan mode for multi-step tasks. Take a moment to read the plan before approving. It is much easier to adjust a plan than to undo completed work. You can modify individual steps or ask for a different approach.

Keep CLAUDE.md Updated

When your project evolves — new features, changed deployment targets, updated brand guidelines — tell Claude to update CLAUDE.md. This ensures every future conversation starts with accurate context.

Use the Super Stack — Do Not Manually Search

Need documentation? Ask Claude. Need to check what is trending? Ask Claude. Need to create a design? Ask Claude. Your MCP servers give Claude direct access to search, docs, GitHub, social media, Canva, and more. Stop switching between browser tabs.

Tell Claude to Remember Important Preferences

When you discover a preference — a design style, a library you like, a deployment pattern — say "remember this." Claude will store it and apply it automatically in future conversations.
